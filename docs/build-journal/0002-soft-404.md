# Build Journal — 0002: Soft-404, end to end

**Date:** 2026-06-30

## Where this began

The plan always called for one feature to travel the entire workflow — idea to live site — before trusting that workflow with bigger work. The soft-404 fix was the chosen proof, and small on purpose: when you're testing a pipeline, a low-risk change means any breakage has an obvious cause.

The bug: requests for pages that don't exist were returning "200 OK" with no real page, instead of a proper "404 Not Found" — wrong for visitors and for search engines, and a quiet reliability gap.

## What shipped

A dedicated not-found page, built entirely from the project's own design system.

- **The page:** a new `404.astro` using the shared Heading and Link components, rendered as static HTML with zero JavaScript — nothing to load, nothing to break.
- **The root cause:** the site builds fully static with no server layer, so there was simply no `404.html` for the host to serve. The fix was additive — add the page — with no redirect rules and no config changes.
- **The guardrail:** a test that loads the page, checks it against accessibility rules, and confirms it actually returns a 404.

## The loop, end to end

This is what the ticket existed to prove. The change went: a Linear ticket → its own branch → a pull request → the automated quality gate (linting, tests, accessibility, performance, all green) → an accessibility review → a fast-forward merge to main → an automatic production deploy → and finally this entry. Two things confirmed themselves along the way: opening the pull request linked it to its ticket on its own, and merging it closed that ticket on its own.

## What's next

The loop is proven on a trivial change. The real exam is a fuller interactive component, with more moving parts, run through the same path. That's what this unblocks.

## Honest notes

This proves the _pipeline_, not the hard part of the work — a one-file static page is about the easiest thing that can make the trip, and a complex component will surface friction this didn't. One smaller question stays open too: whether opening a pull request moves a ticket to "In Progress" automatically, or still needs a manual nudge — a manual flip this time hid the answer. And the entry you're reading was itself the loop's last missing step, which is a bit recursive but honest: the workflow isn't done until writing it down is part of it.
